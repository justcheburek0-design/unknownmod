# Testing Plan

Этот файл содержит чеклист перед публикацией мода в GitHub. Цель не только проверить сборку, но и поймать вероятные проблемы в логике, конфиге, сетевом поведении и после рестарта сервера.

## 1. Автоматическая проверка

Запусти чистую сборку:

```bash
./gradlew clean build
```

Проверь следующее:

- сборка завершается без ошибок;
- jar появляется в `build/libs/`;
- `sources.jar` тоже собирается;
- в выводе Gradle нет новых предупреждений, связанных с несовместимостью API, mixin-инъекциями или ресурсами.

## 2. Старт сервера

Подними отдельный тестовый Fabric-сервер с чистым миром.

Проверь:

- сервер стартует без `Mixin apply failed`;
- нет ошибок загрузки зависимостей;
- `config/unknown.yml` создаётся автоматически;
- в логах нет ошибок чтения или сохранения конфига;
- команда `/unknown` доступна оператору.

## 3. Базовый функционал

Проверь на 2-3 игроках:

- вход игрока получает анонимное имя;
- выход игрока отправляет кастомное сообщение;
- `/unknown nickname <value>` меняет имя после `/unknown reload` или после следующего обновления профиля;
- `/unknown skin <nickname>` успешно подтягивает скин по нику;
- `/unknown skin texture` работает только если в конфиге уже есть сохранённые `texture/signature`.

## 4. `ghost`-сценарии

Проверь следующие случаи:

- `/unknown ghost <player>` переводит игрока в spectator;
- повторный вызов той же команды возвращает игрока обратно;
- после восстановления игрок получает прежний режим игры;
- убийство игрока оружием с кастомным именем, совпадающим с оригинальным именем жертвы, переводит жертву в `ghost`;
- игрок в `ghost` не участвует в выборе цели для `reveal`.

## 5. `reveal`-сценарии

Проверь:

- `/unknown reveal player <player>` запускает `reveal` на выбранном игроке;
- `/unknown reveal player` без аргумента выбирает случайного подходящего игрока;
- `/unknown reveal status` показывает корректный таймер;
- предупреждение приходит за `warning-minutes` до `reveal`;
- `reveal` автоматически завершается по таймеру;
- `/unknown reveal cancel` отменяет все активные `reveal`;
- `/unknown reveal cancel <player>` даёт выбор среди уже раскрытых игроков и отменяет только выбранного;
- при `revelation.enabled = false` `reveal` не стартует;
- при недостатке игроков событие откладывается, а не падает с ошибкой.

## 6. Персистентность и рестарт

Это один из самых частых источников багов перед релизом.

Проверь:

- `ghost`-состояние переживает рестарт сервера;
- активный `reveal` и его расписание переживают рестарт сервера;
- после рестарта таймеры не уходят в отрицательные значения;
- игроки после рестарта не остаются в неправильном режиме игры;
- таб-лист и имя в чате обновляются корректно после перезапуска.

## 7. Негативные тесты

Проверь, что мод не ломается на плохих данных:

- испорченный `unknown.yml` даёт понятную ошибку в логе;
- несуществующий никнейм для `/unknown skin <nickname>` не ломает конфиг;
- отсутствие сохранённых `texture/signature` корректно блокирует `/unknown skin texture`;
- отключённые зависимости или несовместимая версия Minecraft быстро выявляются при старте, а не в середине игры;
- пустой сервер и выход единственного игрока не вызывают исключений в `reveal`/`ghost` логике.

## 8. Ручная проверка перед публикацией

Перед пушем в GitHub пройди этот список:

- `./gradlew clean build` проходит локально;
- сервер стартует на чистом мире без ошибок;
- проверены команды, конфиг и персистентность после рестарта;
- проверен случай с минимумом игроков и срабатыванием `reveal`;
- проверен случай с ручным и автоматическим `reveal`;
- проверен случай с `ghost` и восстановлением режима;
- в репозиторий не попали лишние артефакты из `build/` или локальных папок;
- обновлены версия и описание, если менялась логика.
